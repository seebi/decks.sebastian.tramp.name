Version Control with git

Wikipedia Quote

Git (/ɡɪt/) is a distributed revision control and source code management (SCM) system with an emphasis on speed. Git was initially designed and developed by Linus Torvalds for Linux kernel development

Distributed vs. Centralized

  • Centralized (Subversion, CVS, …)
    • developers send all changes to a central repository
    • the repository does access control
    • only one version of a repository is available
  • Distributed (git, mercurial, darc, …)
    • every developer has a complete copy of the repository
    • changes can be send to each developer
    • complete history of all files is available locally

Concepts I

  • Repository - the history, different versions, branches and tags of a project, local or remote
  • Working Tree - a materialization of a specific revision in the filesystem, maybe changed by you
  • Commit - a specific revision which can be checkout as a working tree, identified by a hash
  • Remote - a named remote repository, sometimes called upstream repository, identified by a URL and name

Concepts II

  • branch - a separate code line with its own history
    • local branch - in your repository
    • remote branch - in someone else repository (in combination with a remote)
    • remote-tracking branch - a local branch with a link to a remote branch
  • tag - a named revision
  • HEAD - a local tag for the revision which is currently used as working tree

Concepts III

Stash vs. Workspace vs. Index

  • Working Tree - the files and changes you see in your IDE / Editor
  • Index - the changed files you've selected for the new commit / revision
  • Stash - The changed files you've laid aside for later use (e.g. for after the checkout)

Basic Workflow: Simple Changes

After cloning the repository:

  • checkout a specific revision, tag or branch
  • change some files with your IDE / editor
  • add files to the index
  • commit changes to your local repository
  • push revisions to a remote repository
